# 密钥到期机制漏洞审计笔记

## 当前机制流程

1. 启动时：`request('http://sensor.bodyta.com:8080/rcv/login/getSystemTime')` 获取服务器时间 → `nowDate`
2. 启动时：读取 `config.txt` → AES 解密 → 得到 `endDate`（到期时间戳）
3. 所有数据处理入口用 `if (nowDate < endDate)` 做门控

## 漏洞清单

### 漏洞1：服务器不可达时 nowDate=0，永远通过
- `nowDate` 初始值为 0
- 如果 `request` 失败（网络断开、服务器宕机），`nowDate` 保持为 0
- `0 < endDate`（任何正的时间戳）→ 永远为 true → 授权永不过期
- **严重程度**：致命。用户只需断网即可绕过授权

### 漏洞2：nowDate 只在启动时获取一次，不会刷新
- 整个应用生命周期中只调用一次 `getSystemTime`
- 如果应用长期运行（比如跨越到期日），不会重新检查
- **严重程度**：高。长时间运行可绕过到期

### 漏洞3：前端没有任何到期检查和提示
- server.js 第 421 行发送 `date: endDate` 给前端
- 但 Home.jsx 中没有处理 `jsonObject.date` 的逻辑（搜索结果为空）
- 前端不知道授权何时到期，没有任何提示
- **严重程度**：高。用户体验差，且无法提前续期

### 漏洞4：到期后只是停止数据处理，无用户提示
- server.js 中 `if (nowDate < endDate)` 只是跳过数据处理
- 到期后串口数据直接丢弃，前端表现为"数据停了"但不知道为什么
- 没有发送任何 `timeExpires: true` 消息（相关代码被注释掉了，第 513-527 行）
- **严重程度**：中。用户困惑，不知道是设备问题还是授权问题

### 漏洞5：AES 密钥硬编码在源码中
- 前后端都硬编码了 `JIANXINGZHEPSVMC` 密钥
- 使用 AES-ECB 模式（最弱的 AES 模式，无 IV）
- 有一定技术能力的用户可以自行生成密钥
- **严重程度**：中。需要一定逆向能力

## 修复方案

### 后端修复
1. 服务器不可达时：回退到本地时间 `Date.now()`，而非保持 0
2. 添加定时刷新：每 30 分钟重新获取服务器时间
3. 到期后发送明确的 `licenseExpired: true` 消息给前端
4. 即将到期时发送 `licenseWarning` 消息（剩余天数）

### 前端修复
1. 接收 `licenseExpired` 消息后显示全屏遮罩提示
2. 接收 `licenseWarning` 消息后显示顶部警告条
3. 显示剩余天数信息
