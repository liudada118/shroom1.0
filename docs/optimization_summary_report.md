## 3. 详细优化过程

### 3.1. 资产与代码清理

我们首先对项目中的冗余资产和代码进行了彻底清理。

- **废弃文件删除**：通过 `find` 和 `grep` 命令定位并删除了 20 个命名中包含 `copy` 的重复文件、8 个未被任何路由或组件引用的前端组件（如 `DemoBed.jsx`），以及 13 个未在 3D 场景中使用的模型文件（`.glb`/`.gltf`）。同时，一个完整的 `client/src1` 目录被确认为废弃代码并被移除。
- **代码引用修复**：在删除文件后，我们检查并修复了 `client/src/App.jsx` 中对已删除文件的无效引用，避免了应用运行时崩溃。

此阶段的清理工作不仅使代码库更加整洁，还显著减小了项目的整体体积，加快了克隆和部署速度。

### 3.2. 日志系统标准化

为了规范日志输出并提升调试效率，我们对项目中的日志记录方式进行了统一。

- **后端**：在后端（主进程），我们利用 `grep` 统计了所有 `console.log`、`console.warn`、`console.error` 的使用情况，并编写了一个 Python 脚本 (`scripts/replace_console.py`)，将 `server.js`、`index.js` 等核心文件中的 118+ 处 `console` 调用批量替换为项目已有的 `logger.js` 模块。这使得后端日志可以被统一管理，例如输出到文件或进行分级控制。
- **前端**：考虑到前端 `console.log` 主要用于开发调试，我们修改了 `client/vite.config.js`，在 `esbuild` 配置中加入了 `drop: ["console", "debugger"]` 选项。这确保了在执行生产环境构建 (`npm run build`) 时，所有 `console` 和 `debugger` 语句都会被自动移除，保证了生产代码的纯净性。

### 3.3. JavaScript 现代化：拥抱 `let` 与 `const`

项目中存在大量使用 `var` 声明变量的情况，这可能导致变量提升、作用域污染等问题。我们编写并运行了 `scripts/replace_var.py` 脚本，安全地将 `server.js` 和 `openWeb.js` 中的 94 处 `var` 声明替换为 `let` 和 `const`。

- **替换策略**：对于 `require()` 语句和初始化后未被重新赋值的变量，使用 `const`；对于循环变量和会被重新赋值的变量，使用 `let`。
- **问题修复**：在替换过程中，我们发现并修复了 `server.js` 中 `serialport` 变量被重复声明两次的缺陷，消除了潜在的运行时错误。

### 3.4. 依赖项清理与优化

通过分析 `package.json` 和代码引用，我们对项目的依赖进行了精简。

- **移除 `request`**：`request` 是一个早已被社区废弃的包。我们定位到 `server.js` 中使用它获取网络时间的代码，并将其替换为 Node.js 内置的 `http` 模块，消除了一个严重的安全和维护隐患。
- **移除其他未使用依赖**：`electron-squirrel-startup`（项目使用 `electron-builder`，而非 `squirrel`）和 `electron-icon-maker`（一次性开发工具，不应在 `devDependencies` 中）也被识别并移除。
- **保留 `sqlite3`**：虽然 `better-sqlite3` 性能更优，但分析发现 `server.js` 中深度使用了 `sqlite3` 的异步回调模式，迁移成本较高。考虑到本次优化的范围，我们决定保留 `sqlite3`，将迁移作为未来的独立任务。

### 3.5. 内存泄漏修复

通过静态代码分析，我们发现 `client/src/page/home/Home.jsx` 组件中存在一个典型的 React 内存泄漏问题。组件中使用 `setTimeout` 实现了一个节流函数 `thrott`，但缺少在组件卸载时（`componentWillUnmount`）进行清理的逻辑。这会导致组件即使被销毁，其定时器依然可能被触发，相关的闭包也无法被垃圾回收。

我们通过在 `componentWillUnmount` 生命周期方法中添加 `clearTimeout(timer)` 和 `clearTimeout(timer1)`，确保了组件卸载时能彻底清理所有异步任务，从而修复了此内存泄漏问题。

### 3.6. 后端模块化：拆分 `server.js`

`server.js` 作为项目后端的“心脏”，代码量超过 4600 行，承担了过多的职责，可维护性极差。我们采取了渐进式重构策略，首先将其中不依赖全局状态的纯函数提取到独立的模块中。

- **创建 `server/mathUtils.js`**：提取了 `gaussBlur_return`、`press6` 等 10 个用于高斯模糊、插值、分压计算的纯数学和数据处理函数。
- **创建 `server/dbManager.js`**：提取了 `initDb` 和 `genDb` 两个负责数据库文件初始化和连接的函数。
- **创建 `server/index.js`**：作为新模块的统一入口，方便其他文件引用。

通过本次拆分，`server.js` 的代码量减少了 **360 行**，相关函数获得了独立的命名空间，更易于理解、测试和复用。这为未来进一步将 WebSocket 逻辑、串口处理逻辑从 `server.js` 中拆分出来奠定了坚实的基础。
