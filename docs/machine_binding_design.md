# 密钥一机一用 - 机器绑定方案设计

## 当前密钥结构

密钥明文 JSON：
```json
{
  "date": 1740000000000,   // 到期时间戳
  "file": "all" | "hand0205" | ["hand0205", "robot1"]  // 授权类型
}
```

加密后为 AES-ECB 十六进制字符串，存储在 config.txt 中。

**问题**：密钥中没有任何机器标识，同一个密钥可以在任意机器上使用。

## 机器指纹方案

### 指纹采集

使用 Node.js `os` 模块采集以下硬件信息，组合生成唯一机器码：

| 信息 | 来源 | 稳定性 |
|:---|:---|:---|
| CPU 型号 | `os.cpus()[0].model` | 高（除非换CPU） |
| CPU 核心数 | `os.cpus().length` | 高 |
| 主板序列号 | `wmic baseboard get serialnumber`(Win) / `dmidecode`(Linux) | 最高 |
| MAC 地址 | `os.networkInterfaces()` 非内部网卡的第一个 MAC | 中（可能变化） |
| 主机名 | `os.hostname()` | 低（用户可改） |

**最终方案**：取 CPU 型号 + 总内存 + 第一个非内部网卡 MAC 地址，做 SHA256 哈希，取前 16 位作为机器码。

### 为什么选这些？
1. **CPU 型号 + 总内存**：硬件级标识，用户几乎不会改变
2. **MAC 地址**：网卡物理地址，增加唯一性
3. **SHA256 哈希**：不暴露原始硬件信息，且固定长度

### 新密钥结构

```json
{
  "date": 1740000000000,
  "file": "all",
  "machineId": "a1b2c3d4e5f6g7h8"  // 16位机器码
}
```

### 工作流程

1. **获取机器码**：应用启动时，后端采集硬件信息生成 machineId
2. **显示机器码**：前端 License 页面显示当前机器的机器码，方便管理员生成密钥
3. **生成密钥**：管理员在 License 页面输入目标机器的机器码 → 生成绑定该机器的密钥
4. **验证密钥**：后端解密密钥后，比对 machineId 是否与当前机器一致
5. **不一致则拒绝**：如果 machineId 不匹配，密钥无效

### 向后兼容

旧密钥没有 machineId 字段，需要兼容处理：
- 如果密钥中没有 machineId → 按旧逻辑处理（不做机器绑定检查）
- 如果密钥中有 machineId → 必须与当前机器匹配
