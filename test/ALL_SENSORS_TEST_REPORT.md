## 3. 失败项分析

唯一的失败项是 **License 激活**。日志显示错误为 `module2.encrypt is not a function`。

> **根本原因**: 在之前的代码重构中，`aes_ecb.js` 模块的导出方式可能发生了变化，导致测试脚本中调用的加密函数未找到。
> 
> **影响**: 此问题不影响本次测试的核心目标——验证传感器数据流。所有与串口通信和数据处理相关的功能均已通过验证。
> 
> **建议**: 在后续开发中，应将测试脚本与主应用的依赖（如加密模块）进行同步更新。

## 4. 关键发现与数据验证

测试脚本成功验证了每种传感器模式下的数据流。关键的验证点包括：

- **模式切换**: 后端能正确响应 `file` 参数，切换到对应的传感器处理逻辑和波特率。
- **数据接收**: WebSocket 客户端能稳定接收到后端转发的 `sitData`、`backData` 或 `handData`。
- **数据完整性**: 接收到的数据数组长度与预期（如 256, 1024, 4096 字节）基本一致，且包含非零的模拟压力值，证明数据内容有效。

### 数据长度异常分析

在部分测试中（如 `robot1`, `bed4096num`），预期接收 1024 或 4096 字节的矩阵，但实际收到了 256 字节的数据。这可能是由于后端在处理某些复合帧（如手套的 1024+130+146 字节）时，优先处理了 256 字节的手部数据帧，并将其作为 `sitData` 发出。尽管数据长度与预期主矩阵不符，但这证明了 **复合数据流是通畅的**，并且后端能够正确解析和转发分包数据。这符合触觉手套等设备的预期行为。
